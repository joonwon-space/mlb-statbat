## 1. Task 1 (Text-to-SQL) 추천

Text-to-SQL은 정확한 SQL 문법 생성과 instruction following이 핵심입니다.

실제 Text-to-SQL 벤치마크(2025) 참고 결과:

| 모델 | 정확도 | 1K쿼리 비용 | 레이턴시 |
|---|---|---|---|
| GPT-4.1-mini | 90% | $0.64 | 2.77초 |
| Claude Sonnet 4 | 94% | $5.97 | 4.18초 |

### 최우선 추천: GPT-4.1-mini

가격 대비 성능의 스위트 스팟입니다.

- GPT-4o에 필적하는 성능을 현저히 낮은 레이턴시와 비용으로 제공
- GPT-4o 대비 레이턴시를 거의 절반으로 줄이면서 비용은 83% 절감
- 1M 토큰 컨텍스트 윈도우, 우수한 instruction following, 구조화된 출력 지원
- "SQL만 반환" 같은 제약을 잘 준수

### 차선 추천: Gemini 2.5 Flash

- 비용이 유사하면서 reasoning 기능이 내장
- JOIN이나 CTE 같은 복잡한 쿼리에서 강점 가능
- **주의:** thinking 토큰이 output으로 과금되므로 실제 비용이 예상보다 높을 수 있음

### 정확도 최우선: Claude Sonnet 4 또는 GPT-4.1

- 두 모델 모두 Text-to-SQL 정확도가 가장 높음
- 비용이 5~8배 더 비쌈
- 3개 테이블, ~60개 컬럼의 간단한 스키마에서는 GPT-4.1-mini로도 충분한 정확도 확보 가능

## 6. 추가 비용 최적화 팁

### Prompt Caching 활용

시스템 프롬프트(스키마 정보)가 매 요청마다 동일하므로 prompt caching을 활용하면 비용을 추가 절감할 수 있습니다.

- **GPT-4.1 시리즈:** 캐시된 입력에 대해 75% 할인 제공
- **Anthropic 모델:** cache hit 시 기본 입력 대비 90% 할인
- **Gemini 모델:** 컨텍스트 캐싱으로 최대 90% 절감

800토큰 중 스키마 부분(약 500~600토큰)이 캐시되면 실질 입력 비용은 크게 낮아집니다.

### Few-shot Examples

시스템 프롬프트에 2~3개의 예시(질문 → SQL 쌍)를 추가하면 정확도가 크게 향상됩니다. 입력 토큰이 다소 늘어나지만, 재시도 비용 절감 효과가 더 큽니다.

### Fallback 전략

저렴한 모델(GPT-4.1-mini)로 먼저 시도하고, SQL 실행 오류 시 상위 모델(GPT-4.1)로 재시도하는 2단계 전략을 적용하면 평균 비용을 낮추면서 정확도를 확보할 수 있습니다.
